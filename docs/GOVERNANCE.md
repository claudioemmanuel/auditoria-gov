# Governance Policy — Source Integrity, Veracity & Compliance

**Version:** 1.1
**Effective date:** 2026-03-11
**Review cycle:** Quarterly

---

## 1. Purpose

AuditorIA Gov produces public risk signals involving political agents and state suppliers. Because these signals can influence public perception and oversight actions, the platform must guarantee:

1. **Source authority** — data comes from legally authoritative origins.
2. **Veracity** — every source carries a transparent, reproducible quality score.
3. **Reproducibility** — any citizen or auditor can trace a signal back to its raw API response.
4. **LLM independence** — risk scores are deterministic; LLMs are explanatory only.

This policy formalises these guarantees at CGU/TCU/MPF audit standards.

---

## 2. Source Classification

### 2.1 Government Domain Whitelist

All outbound HTTP requests pass through `shared/connectors/domain_guard.py`. Only domains matching these Brazilian government TLDs are allowed without exception:

| TLD | Scope |
|-----|-------|
| `.gov.br` | Federal, state, and municipal executive |
| `.jus.br` | Judiciary |
| `.leg.br` | Legislative (Senado, Câmara) |
| `.mil.br` | Military |
| `.mp.br` | Ministério Público |
| `.def.br` | Defensoria Pública |

### 2.2 Controlled Exceptions

Non-government sources require formal approval with a time-limited exception:

| Domain | Justification | Max Veracity | Approved | Review By |
|--------|--------------|-------------|----------|-----------|
| `api.queridodiario.ok.org.br` | Open Knowledge Brasil — municipal gazette aggregation; no official federal API exists | 0.85 | 2026-03-04 | 2026-09-04 |

**Nota sobre o score do Querido Diário (0.435 — "low"):** O score reflete que a fonte não é hospedada em domínio governamental e não possui API oficial documentada pelo governo. A exceção é mantida porque: (1) o Querido Diário é o único agregador nacional de diários municipais disponível; (2) Open Knowledge Brasil é uma organização estabelecida e auditável; (3) sinais baseados exclusivamente nessa fonte recebem penalidade de veracidade na composição do score final. Nenhum sinal crítico depende exclusivamente desta fonte.

Exceptions are defined in `DOMAIN_EXCEPTIONS` within `domain_guard.py` and enforced at the HTTP client layer.

### 2.3 Adding a New Source

To add a new data source:

1. Verify the API domain against the whitelist.
2. If non-government, submit a `DomainException` with justification, max veracity ceiling, and 6-month review date.
3. Create a `SourceVeracityProfile` in `shared/connectors/veracity.py`.
4. Update this document with the new source's veracity table entry.
5. Add tests verifying domain validation and profile completeness.

## 3. Veracity Scoring Methodology

Each source receives a weighted composite score:

| Criterion | Weight | Description |
|-----------|--------|-------------|
| Government domain | 40% | Is the API hosted on a government TLD? |
| Legal authority | 25% | Does the source have legal mandate to publish this data? |
| Public availability | 15% | Is the data freely accessible without restricted credentials? |
| Official API documented | 10% | Does the source publish API documentation? |
| Metadata & traceability | 10% | Does the response include timestamps, IDs, and audit fields? |

### 3.1 Labels

| Label | Threshold | Meaning |
|-------|-----------|---------|
| `official` | >= 0.95 | Government source with full legal authority |
| `high` | >= 0.85 | Government source or well-documented exception |
| `acceptable` | >= 0.70 | Approved exception with known limitations |
| `low` | < 0.70 | Should not be used without escalation |

### 3.2 Per-Source Veracity Table

| Connector | Domain Tier | Composite Score | Label |
|-----------|------------|----------------|-------|
| Portal da Transparência | government | ~0.995 | official |
| Compras.gov.br | government | ~0.97 | official |
| ComprasNet Contratos | government | ~0.957 | official |
| PNCP | government | ~0.985 | official |
| TransfereGov | government | ~0.97 | official |
| Câmara dos Deputados | government | ~0.99 | official |

Profiles are defined in `shared/connectors/veracity.py` and exposed via `GET /public/sources`.

## 4. LLM Usage Policy

### 4.1 Principle

**LLMs are explanatory only.** They never participate in scoring, classification, or risk detection.

### 4.2 Enforcement

- The `@explanatory_only` decorator on `explain_signal()` (and any future LLM function) asserts that:
  - The return type is `str` (markdown explanation text).
  - A `TypeError` is raised if the function returns a number, dict, or any non-string type.
  - Every invocation is logged as an audit event.
- Risk signals are generated by deterministic typology rules in `shared/typologies/`.
- LLM outputs are stored in `explanation_md` — a display-only field that does not affect `severity`, `confidence`, or `completeness_score`.

### 4.3 Providers

| Provider | Purpose | Fallback |
|----------|---------|----------|
| OpenAI | Explanations + embeddings | NoOp (deterministic template) |
| Anthropic | Explanations | NoOp (deterministic template) |
| NoOp | Default when LLM disabled | — |

Setting `LLM_PROVIDER=none` in the environment disables all LLM calls and uses Jinja2 templates.

## 5. Compliance Automation

### 5.1 Weekly Audit Task

`check_source_compliance` runs every Monday at 06:00 UTC:

1. **Domain validation** — verifies all connector URLs against the whitelist.
2. **Exception expiry check** — warns 30 days before a `DomainException.review_by` date.
3. **API health probe** — HTTP HEAD to each source; flags unreachable APIs.
4. **Registry update** — writes `compliance_status` (`ok` | `warning` | `violation`) and `last_compliance_check_at` to `coverage_registry`.

### 5.2 Compliance Statuses

| Status | Meaning |
|--------|---------|
| `ok` | Domain whitelisted, API reachable |
| `warning` | API unreachable or exception expiring soon |
| `violation` | Domain not in whitelist (should never happen in production) |

## 5b. Data Quality Monitoring

A dedicated data quality endpoint (`GET /internal/data-quality`) continuously monitors the health of all ingested sources:

- **Per-source metrics**: entity count, freshness lag (hours since `last_success_at`), veracity score, compliance status
- **Cross-source overlap histogram**: entity distribution by number of sources (1 source vs. 2 vs. 3+), revealing coverage depth and deduplication quality
- **Week-over-week delta alerts**: any source whose weekly entity contribution drops >20% triggers a warning — indicating ingestion failures, API changes, or data drift
- **LGPD scoping audit**: person-type entity search is verified to exclude non-public-servant sources via `EntityRawSource.source_connector` join

These metrics complement the existing `CoverageRegistry` tracked per connector and are visible in the admin data quality dashboard (`/coverage/quality`).

## 6. Reproducibility Guarantees

Every risk signal can be traced back to its raw source data:

```
RiskSignal
  → SignalEvent → Event → EventRawSource → RawSource (raw_data JSONB)
  → SignalEntity → Entity → EntityRawSource → RawSource
```

The `GET /signal/{id}/provenance` and `GET /case/{id}/provenance` endpoints expose this chain publicly.

## 7. Legal Risk Mitigations

| Risk | Mitigation |
|------|-----------|
| Accusation without evidence | All signals include "indicador estatístico, não acusação" disclaimer |
| Data from unauthorized source | Domain guard blocks non-whitelisted domains at HTTP layer |
| LLM hallucination affecting scores | `@explanatory_only` decorator; LLM never touches scoring |
| Stale data leading to false signals | Coverage registry tracks freshness; stale sources flagged |
| LGPD violation (CPF exposure) | CPF hashing via `shared/utils/hashing.py`; no raw CPF persistence |
| Exception source without review | Weekly compliance task warns 30 days before expiry |

---

## 8. Public Transparency

The `GET /public/sources` endpoint exposes:

- All data sources with their base URLs and government/exception classification.
- Per-source veracity scores (all 5 criteria + composite).
- The domain whitelist and controlled exceptions list.
- Freshness, item counts, and compliance status.
- Policy version for tracking changes.

This ensures any citizen, journalist, or auditor can verify the platform's data provenance.
